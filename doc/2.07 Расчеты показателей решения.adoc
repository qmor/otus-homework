=== Расчеты показателей решения

[cols="3^8"]
|===
|Требование | расчет | Пояснение
|Низкая временная задержка при передаче тескового сообщения (менее 500 мс)
|Среднее время: 300–400 мс
|Для 5000 DAU -> ~1000 сообщений/мин -> 16.7 сообщений/сек. Нагрузка не очень большая.
|Возможность загрузки медиаконтента (в т.ч. документов, архивов) до 2 гигабайт - время загрузки до 15 минут
|2 ГБ за 15 мин = ~2.25 МБ/с
|Требуется высокая пропускная способность+rate limiter+балансировка нагрузки и возможно CDN.
|Задержка при передаче изображения < 1 сек
|Для изображений меняем разрешение перед сохранением, что не превышало 2048  - полагаем что средний максимальный размер изображения 5Мб - при требовании задержки при передаче изображения 1000 мс имеет необходимую скорость канала > 5 мб сек
|Требуется балансировка нагрузки/кеширование популярных изображений (картинка в групповом чате)
|Воспроизведение видео < 1 сек после play
|Необходимо перекодировать видео в несколько битрейтов, которые дать возможность переключать пользователю, либо которые выбираются автоматически в зависимости от возможностей CDN И канала пользователя. Необходим стриминг с небольшим буфером на клиенте - заполняющийся по мере подгрузки видео. H.264 720p в 30 fps в стандартном качестве использует примерно 4 Mbps
|Требуется использование h264 как изначально поддерживающего стриминг, возможно использование протокола quic/http3 для оптимального стриминга с низкой задержкой, балансировка + CDN
|Надежность 99.9 доступность, RPO 15 минут для истории прямых чатов, RPO 5 мин для истории групповых чатов. RPO 30 минут для медиа.
|99.9% = 8.76 часа в год сбоя
|Требуется обеспечить хранение медиа
|DAU(5000) × (1 изображение/день × 5МБ + 1 видео/неделю × 50МБ/7 + 1 документ/день × 10МБ) = ~110.7 ГБ/день = ~40.4 ТБ/год
|===


=== Компоненты влияющие на производительность

[cols="3^8"]
|===
|Компонент|влияние на производительность|Комментарий
|API Gateway|Высокое|Входная точка - должен обрабатывать самый большой поток запросов
|Сервис авторизации|Высокое|Требует высокой доступности
|Сервис чатов (директ/группы)|Высокое|Хранит и передаёт сообщения. Должен обрабатывать 100+ сообщений/сек.
|Сервис медиа (S3 + CDN)|Высокое|Обработка файлов до 2 ГБ — требует высокой пропускной способности.
|Kafka (брокер сообщений)|Высокое|Асинхронная доставка уведомлений, межсервисное асинхронное взаимодействие, критичен для масштабируемости.
|Redis-кэш|Высокое|Снижает нагрузку на S3 при публикации медиа. Но на занятии уточнили что редис это не то решение что подходит для медиа - поэтому используем CDN
|===

*Расчет количества запросов к системе

*Бизнес-событие: Отправка текстового сообщения*

.1 запрос к API Gateway (POST /api/chats/{chat_id}/messages)
.1 проверка токена в Auth Service (gRPC)
.1 запись сообщения в PostgreSQL (Chat Service)
.1 публикация события в Kafka (MessageCreatedEvent)
.1 обработка в Notification Service (подписка на Kafka)
.1 индексация в Search Service (подписка на Kafka)

*Итого: 1 бизнес-событие -> 6 технических запросов*

*Бизнес-событие: Загрузка медиа (2 ГБ)*

.1 запрос к API Gateway (POST /api/media/upload/init)
.1 запрос к S3 (генерация presigned URL)
.1 загрузка файла в S3 (multipart upload, ~1000 частей)
.1 событие в Kafka (MediaUploaded)
.1 запись метаданных в PostgreSQL
.1 индексация в Search Service

*Итого: 1 бизнес-е -> 5-6 технических запросов + 1000 HTTP запросов к S3*

*Бизнес-событие: Аутентификация*

.1 запрос к API Gateway (POST /api/auth/login)
.1 проверка в Auth Service (gRPC)
.1 запрос к User Service (gRPC) -> PostgreSQL
.1 запрос к LDAP (внешняя интеграция)
.1 генерация JWT

*Итого: 1 бизнес-е -> 5 технических запросов*

Общая оценка нагрузки

[cols="3^8"]
|===
|Параметр | Расчет | Значение
|DAU|5000 пользователей|Базовое значение
|Активность|20% пользователей онлайн одновременно|1000 пользователей онлайн
|Сообщений в минут|1000 сообщений/мин|~16.7 сообщений/сек
|Запросов к API Gateway|~100 сообщений/сек × 7 запросов|~700 запросов/сек (пик)
|Запросов к Kafka|~100 сообщений/сек × 2 события|~200 событий/сек
|Запросов к PostgreSQL|~100 сообщений/сек × 2 (чаты + пользователи)|~200 запросов/сек
|Запросов к S3|~5 загрузок/мин × 1000 частей|~83 запросов/сек (пик при загрузке медиа)
|===

Вывод: Архитектура способна выдержать текущую нагрузку с запасом 2-3×. При росте нагрузки до 15000 DAU потребуется горизонтальное масштабирование сервисов и увеличение количества реплик.
