=== Схема архитектуры решения

Оценка готовности архитектуры
[cols="2^8"]
|===

|Требование | Готовность | Пояснение

|Задержка < 500 мс (текст)| Высокая | Система использует WebSocket + REST + gRPC должно обеспечить требуемое время
|Загрузка медиа до 2 ГБ | Частично готово | S3 поддерживает большие файлы, но требуется multipart upload + CDN для балансировки
|Задержка загрузки изображения < 1 сек | Высокая | С помощью предварительной подготовки изображения и CDN
|Воспроизведение видео < 1 сек | Не готово | Требуется реализовать стриминг  + предварительная загрузка
|Масштабируемость (5000 DAU)| Высокая| Микросервисы и Kafka - масштабируются горизонтально
|99.9% доступность| Высокая | С помощью кластеризации, балансировки, мониторинга - возможно
|RPO 5–30 мин | Необходимо доработать | Требуется резервное копирование 


Масштабирование в текущей архитектуре
[cols="2^8"]
|===
|Компонент | Масштабируемость 
|Api-gateway| Горизонтальное масштабирование путём добавления реплик, балансировщик распределяет по репликам
|Сервис чатов| CQRS + репликация PostgreSQL + шардинг по chat_id
|Сервис медиа| Горизонтальное масштабирование + S3 + CDN + Redis кэш
|Сервис авторизации| Горизонтальное масштабирование + кэширование JWT в Redis
|Сервис уведомлений| Горизонтальное масштабирование + Kafka
|Сервис поиска| Elasticsearch кластер (3+ ноды) + шардинг
|Брокер сообщений (Kafka)| Кластер из 3+ брокеров + разделение топиков на партиции
|БД пользователей| Репликация PostgreSQL (primary-replica) + чтение с реплик
|Redis (кэш)| Redis Cluster (3+ ноды) + репликация

===

Механизмы отказоустойчивости

[cols="2^8"]
|===
|Компонент | Механизм | Описание

|Client <> Server | Rate Limiter | Ограничение запросов на API Gateway (например, 100 req/sec на IP) для защиты от DoS

|Сервис <> Сервис | Retry + Backoff | При сетевых ошибках - повторные запросы с экспоненциальной задержкой

|Сервис <> Сервис | Circuit Breaker | Если сервис недоступен > X раз, обрываем цепочку и возвращаем кеш/ошибку

|Асинхронная обработка | Async Processing | Уведомления, индексация, аналитика - только через Kafka. Если Kafka перегружен - сообщения в очереди (DLQ)

|База данных | Connection Pooling | Использование PgBouncer для предотвращения исчерпания соединений


|PostgreSQL | Streaming Replication | Репликация в реальном времени с автоматическим failover

===

Что переводим на асинхронную обработку

* Уведомления (push, email)
* Индексация сообщений в поиске (Elasticsearch)
* Обработка медиа (конвертация, создание превью)
* Аналитика и логирование
* Отправка OTP через почтовый сервер

===

Где добавляем rate-limiter, retry, circuit breaker

*Rate Limiter:*
** API Gateway - на уровне входа (по IP, по пользователю, по endpoint)
** Сервис авторизации - защита от брутфорса (ограничение попыток входа)

*Retry:*
** API Gateway -> Сервисы (при временных сетевых ошибках)
** Сервисы -> Базы данных (при временных проблемах с соединением)
** Сервисы -> Внешние API (LDAP, Push-сервисы)

*Circuit Breaker:*
** API Gateway -> Сервис авторизации (если сервис недоступен)
** Сервис чатов -> Kafka (если брокер перегружен)
** Сервис уведомлений -> Push-сервисы (если сервис недоступен)
