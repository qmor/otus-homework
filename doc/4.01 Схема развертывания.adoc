=== Схема развертывания

==== Общий подход

Система развертывается в Kubernetes кластере с использованием подхода Infrastructure as Code (IaC). Это обеспечивает воспроизводимость, версионирование инфраструктуры и возможность быстрого восстановления.

==== 1. Infrastructure as Code (IaC)

===== Инструменты IaC

**Для управления Kubernetes:**
- **Helm** - управление релизами приложений и зависимостями
- **Kustomize** - кастомизация конфигураций для разных окружений
- **ArgoCD** - GitOps подход к развертыванию приложений

**Для управления инфраструктурой:**
- **Ansible** - конфигурирование серверов, установка Kubernetes и системного ПО

===== Развертывание инфраструктуры

**Базовая инфраструктура (Ansible):**
- Установка и настройка Kubernetes кластера (kubeadm/k3s)
- Настройка сетевых правил и firewall
- Установка и настройка PostgreSQL
- Установка и настройка Redis
- Установка MinIO для хранения медиа-файлов
- Установка HashiCorp Vault для секретов
- Установка мониторинга и логирования (Prometheus, Grafana, Loki)

**Хранение конфигураций:**
- Все Ansible playbooks в Git репозитории
- Inventory файлы для разных окружений
- Версионирование всех изменений инфраструктуры
- Использование Ansible Vault для секретов

===== Архитектура инструментов IaC

**Разделение ответственности:**

**Ansible используется для:**
- Установки и настройки Kubernetes кластера (self-hosted)
- Конфигурирования серверов перед созданием кластера
- Установки системного ПО (PostgreSQL, Redis, MinIO, Vault)
- Настройки сетевых правил и firewall
- Установки и настройки мониторинга

**Helm/Kustomize используются для:**
- Управления приложениями в Kubernetes
- Упаковки микросервисов в Helm charts
- Кастомизации конфигураций для разных окружений

**ArgoCD используется для:**
- GitOps подхода к развертыванию
- Автоматической синхронизации с Git
- Отслеживания состояния приложений
- Rollback при проблемах

===== Развертывание прикладного ПО

**Прикладное ПО (Helm Charts):**
- Каждый микросервис упакован в Helm chart
- Helm charts хранятся в отдельном репозитории
- Версионирование Helm charts соответствует версиям приложений
- Зависимости между сервисами управляются через Helm

**Структура Helm chart:**
```
service-name/
├── Chart.yaml
├── values.yaml
├── values-dev.yaml
├── values-staging.yaml
├── values-prod.yaml
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    ├── configmap.yaml
    └── secret.yaml
```

===== Обновление ПО

**Стратегия обновлений:**
- Rolling updates для stateless сервисов
- Canary deployments для критических изменений
- Blue-green deployments для крупных релизов
- Database migrations выполняются отдельно от приложения

**Процесс обновления:**
1. Изменение кода в feature branch
2. Создание Pull Request и Code Review
3. Слияние в master branch
4. Автоматический запуск CI/CD пайплайна
5. Сборка Docker образа
6. Запуск тестов
7. Пуш образа в registry
7. Обновление Helm chart
8. ArgoCD автоматически применяет изменения


**Откат изменений:**
- Автоматический rollback при неудачном деплое
- Хранение предыдущих версий Helm releases
- Возможность быстрого отката через ArgoCD UI

==== 2. Управление качеством решения

===== Подходы для повышения надежности

**Автоматизация:**
- Автоматическое тестирование на каждом этапе CI/CD
- Автоматическое развертывание в staging окружение
- Автоматические health checks и readiness probes
- Автоматическое масштабирование (HPA)
- Автоматическое восстановление при сбоях (liveness probes)

**Регламентируемые процедуры:**
- Ручное тестирование в staging перед продакшеном
- Code Review для всех изменений
- Плановые проверки безопасности
- Регулярное тестирование DR (Disaster Recovery)
- Обновление зависимостей с проверкой совместимости



===== Что выполняется регламентом, что автоматизируется

**Автоматизируется:**
- Сборка и тестирование кода
- Развертывание в dev и staging
- Создание бэкапов
- Проверка работоспособности сервисов
- Масштабирование при нагрузке
- Восстановление после сбоев
- Сбор метрик и логов

**Выполняется регламентом:**
- Code Review
- Ручное тестирование в staging
- Утверждение релизов
- Плановое обновление зависимостей
- Аудит безопасности
- Тестирование DR процедур
- Обучение персонала

==== 3. Эффективность эксплуатации решения

===== Работа с разными окружениями

**Окружения:**
- **Dev** - разработка, быстрые итерации, минимальные ресурсы
- **Staging** - предпродакшн, полное тестирование, ресурсы как в проде
- **Production** - продакшн, высокая доступность, масштабирование

**Изоляция окружений:**
- Отдельные Kubernetes namespaces
- Отдельные базы данных
- Отдельные S3 buckets
- Разные домены/URL
- Разные конфигурации

**Доставка изменений:**
- Dev → Staging: автоматически при merge в master
- Staging → Production: вручную после успешного тестирования
- Использование GitOps для отслеживания состояния

===== Применение контейниризации

**Docker:**
- Каждый микросервис в отдельном контейнере
- Multi-stage builds для оптимизации размера
- Использование официальных базовых образов
- Регулярное обновление базовых образов для безопасности

**Kubernetes:**
- Оркестрация контейнеров
- Управление ресурсами (CPU, memory limits)
- Service discovery и load balancing
- автоматический перезапуск сервисов
- Rolling updates и rollbacks



==== 4. Испытания

===== Автоматизация тестирования

**Типы тестов:**
- **Unit tests** - тестирование отдельных функций и классов
- **Integration tests** - тестирование взаимодействия сервисов
- **E2E tests** - тестирование пользовательских сценариев
- **Contract tests** - тестирование API контрактов между сервисами

**Инструменты:**
- **JUnit** - unit и integration tests
- **Testcontainers** - интеграционные тесты с реальными БД
- **Cypress** - E2E тесты веб-интерфейса
- **Pact** - contract testing
- **K6** - нагрузочное тестирование

**Запуск тестов:**
- Unit tests на каждом commit
- Integration tests на каждом PR
- E2E tests на каждом merge в master
- Contract tests при изменении API
- Load tests перед релизами

===== Нагрузочное тестирование

**Сценарии тестирования:**
- **Базовая нагрузка** - 1000 concurrent users
- **Пиковая нагрузка** - 5000 concurrent users (DAU)
- **Стресс-тестирование** - 10000+ concurrent users
- **Endurance testing** - длительная нагрузка 24 часа
- **Spike testing** - резкое увеличение нагрузки

**Метрики для мониторинга:**
- Response time (p50, p95, p99)
- Throughput (requests per second)
- Error rate
- CPU и memory utilization
- Database connection pool usage
- Network I/O

**Инструменты:**
- **K6** - скриптовое нагрузочное тестирование
- **Gatling** - нагрузочное тестирование с записью сценариев
- **Locust** - распределенное нагрузочное тестирование
- **JMeter** - комплексное нагрузочное тестирование

**Процесс:**
1. Разработка тестовых сценариев
2. Запуск на staging окружении
3. Анализ результатов и выявление bottleneck'ов
4. Оптимизация и повторное тестирование
5. Утверждение результатов перед релизом

==== 5. Observability

===== Мониторинг

**Метрики:**
- **Business metrics** - активные пользователи, сообщения, чаты
- **Application metrics** - request rate, error rate, latency
- **Infrastructure metrics** - CPU, memory, disk, network
- **Kubernetes metrics** - pods, nodes, deployments

**Инструменты:**
- **Prometheus** - сбор и хранение метрик
- **Grafana** - визуализация и дашборды
- **Alertmanager** - управление алертами

**Ключевые дашборды:**
- Обзор системы 
- Производительность API 
- Состояние сервисов 
- Базы данных 
- Kubernetes кластер 

===== Логирование

**Структура логов:**
- Структурированные логи в json
- Correlation ID для трассировки запросов
- Уровни логирования: DEBUG, INFO, WARN, ERROR
- Контекстная информация (user_id, request_id, etc.)

**Инструменты:**
- **Loki** - агрегация и хранение логов
- **Promtail** - сбор логов
- **Grafana** - поиск и визуализация логов

**Типы логов:**
- Application logs
- Access logs (API Gateway)
- Error logs
- Audit logs
- Security logs

===== Трассировка

**Distributed Tracing:**
- **Jaeger** или **Tempo** - распределенная трассировка
- OpenTelemetry для сбора трасс
- Визуализация пути запроса через микросервисы
- Идентификация медленных операций

**Метрики трассировки:**
- Latency по каждому сервису
- Количество вызовов между сервисами
- Error rate по операциям
- Dependency graph сервисов
