==== PostgreSQL (Реляционная БД)

*Назначение:* Храним здесь сообщения, пользователей, чаты и события

*Масштабирование:*
* Можно добавить больше мощности (CPU, RAM)
* Можно разделить данные на несколько серверов (шардирование)
* Можно добавить реплики для чтения

*Производительность:*
* Индексы ускоряют поиск
* Разделение данных по дате ускоряет запросы
* PgBouncer управляет соединениями

*Высокая доступность:*
* Репликация копирует данные на другие серверы
* Patroni автоматически переключается на рабочий сервер при сбое
* Регулярные бэкапы

*Ограничения:*
* Максимальный размер базы: 32 ТБ
* Максимум 1000 соединений
* Время отклика: < 100 мс

==== S3 / MinIO (Объектное хранилище)

*Назначение:* Храним медиа-файлы, бэкапы, статические ресурсы

*Производительность:*
* Загрузка файла частями (multipart upload)
* Прямая загрузка от клиента через presigned URL

*Высокая доступность:*
* S3 очень надежный (99.99% доступности)
* Данные хранятся в нескольких копиях

*Отказоустойчивость:*
* Versioning сохраняет все версии файлов
* Можно восстановить удаленные файлы
* Автоматическое перемещение старых файлов в архив

*Ограничения:*
* Максимальный размер файла: 5 ТБ
* Без ограничений на количество файлов

==== Redis Cluster (In-memory БД)

*Назначение:* Сессии пользователей, кэш, rate limiting

*Масштабирование:*
* Можно добавить больше памяти
* Redis Cluster автоматически распределяет данные по серверам

*Производительность:*
* Очень быстро: GET/SET < 1 мс
* До 100K операций в секунду на один сервер

*Высокая доступность:*
* Репликация копирует данные на другие серверы
* Автоматическое переключение при сбое
* Сохранение данных на диск (RDB и AOF)

*Ограничения:*
* Максимальный размер ключа: 512 МБ
* Ограничено объемом памяти

==== Elasticsearch (Поисковая БД)

*Назначение:* Полнотекстовый поиск, агрегации

*Масштабирование:*
* Можно добавить больше серверов в кластер
* Автоматическое распределение данных

*Производительность:*
* Индексация: до 10K документов в секунду
* Поиск: < 100 мс для простых запросов

*Высокая доступность:*
* Репликация копирует данные
* Автоматическое переключение при сбое
* Регулярные снапшоты в S3

*Ограничения:*
* Максимальный размер документа: 100 МБ
* Требует много памяти

==== Kafka (Брокер сообщений)

*Назначение:* Асинхронные уведомления, обмен сообщениями между сервисами

*Масштабирование:*
* Можно добавить больше брокеров
* Разделение данных на партиции

*Производительность:*
* До 100K сообщений в секунду
* Задержка: < 100 мс

*Высокая доступность:*
* Каждое сообщение копируется на 3 брокера
* Гарантия доставки сообщений

*Ограничения:*
* Максимальный размер сообщения: 1 МБ
* Сообщения хранятся 7 дней

==== Резервные системы

*Active-Passive:* Основной регион работает, резервный готов к запуску. PostgreSQL Primary-Replica, S3 cross-region replication

*Warm Standby:* Отдельные кластеры для Elasticsearch и Redis для восстановления и аналитики

==== Тестирование восстановления

*Процедура:*

1. **Еженедельно:** восстанавливаем PostgreSQL, проверяем целостность
2. **Ежемесячно:** полное переключение на резервную систему
3. **Квартально:** симуляция катастрофы, восстановление из бэкапа

*RTO (время восстановления):* PostgreSQL < 1 час, S3 < 5 мин, Redis < 10 мин, Elasticsearch < 2 часа

*RPO (потеря данных):* PostgreSQL < 15 мин, S3 < 1 мин, Redis < 5 мин, Elasticsearch < 1 день

==== Мониторинг бэкапов

*Alerts:* бэкап не создан, бэкап поврежден, время восстановления слишком долгое

*Дашборды:* статус бэкапов, размер, время создания
