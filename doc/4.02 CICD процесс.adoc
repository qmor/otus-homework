=== CI/CD процесс

==== Общий подход

CI/CD пайплайн автоматизирует процесс доставки кода от разработчика до продакшена. Используется GitOps подход с ArgoCD для управления развертываниями в Kubernetes.

==== 1. CI Pipeline (Continuous Integration)

===== Стадии CI пайплайна

**1. Checkout**
- Клонирование репозитория
- Проверка ветки и коммита

**2. Linting**
- Статический анализ кода SonarQube
- Проверка Dockerfile (hadolint)
- Проверка Kubernetes manifests (kubeval)

**3. Unit Tests**
- Запуск unit тестов для всех сервисов
- Покрытие кода тестами (минимум 80%)
- Генерация отчетов о покрытии

**4. Security Scanning**
- Сканирование зависимостей на уязвимости (Snyk, Dependabot)
- Сканирование Docker образа (Trivy)
- Проверка секретов в коде (gitleaks)

**5. Build**
- Сборка Docker образов для каждого сервиса
- Multi-stage builds для оптимизации размера
- Тегирование образов (commit hash, branch name)

**6. Push to Registry**
- Пуш Docker образов в container registry
- Использование semantic versioning
- Хранение образов с тегами для отката

**7. Integration Tests**
- Запуск integration тестов с Testcontainers
- Тестирование взаимодействия между сервисами
- Тестирование базы данных

**8. Contract Tests**
- Проверка API контрактов (Pact)
- Валидация совместимости сервисов

===== Инструменты CI

** GitLab CI / Jenkins:**
- Автоматический запуск на каждый push и PR
- Параллельное выполнение тестов
- Кэширование зависимостей для ускорения
- Артефакты для хранения результатов тестов


==== 2. CD Pipeline (Continuous Deployment)

===== Стратегии развертывания

**Dev окружение:**
- Ручное развертывание из MR
- Использование feature flags для новых функций


**Staging окружение:**
- Автоматическое развертывание при merge в master branch
- Полное тестирование перед продакшеном
- Использование реальных данных (анонимизированных)

**Production окружение:**
- Ручное, после успешного тестирования в staging
- Canary deployment для критических изменений
- Мониторинг после деплоя

===== GitOps с ArgoCD

**Принципы GitOps:**
- Вся конфигурация в Git
- Желаемое состояние описано в Git
- Автоматическая синхронизация с кластером
- Отклонения от желаемого состояния детектируются

**Структура GitOps репозитория:**
```
gitops-repo/
├── apps/
│   ├── auth-service/
│   │   ├── values-dev.yaml
│   │   ├── values-staging.yaml
│   │   └── values-prod.yaml
│   ├── chat-service/
│   └── ...
├── clusters/
│   ├── dev/
│   ├── staging/
│   └── prod/
└── README.md
```

**Процесс развертывания:**
1. Разработчик создает PR в service repo
2. CI пайплайн собирает и тестирует
3. После merge обновляется image tag в GitOps repo
4. ArgoCD обнаруживает изменение
5. ArgoCD автоматически применяет изменения в кластер
6. Health checks проверяют состояние
7. При проблемах автоматический rollback

==== 3. Оперативная доставка релизов

===== Ветка разработки

**Git Flow:**
- `master` - продакшн код
- `feature/*` - новые функции
- `hotfix/*` - срочные исправления
- `release/*` - подготовка релиза

**Процесс релиза:**
1. Разработка в feature ветках
2. Merge в release/* через Pull Request
3. Тестирование в dev окружении
4. Финальное тестирование в staging
5. Merge release в master
6. Тегирование релиза (semver v1.0.0)
7. Развертывание в production

===== Feature Flags

**Использование feature flags:**
- Новые функции скрыты за флагами
- Включение функций без деплоя
- A/B тестирование
- Быстрый rollback проблемных функций



==== 4. Обновления без простоев

===== Rolling Updates

**Принцип работы:**
- Постепенная замена старых Pods на новые
- Health check каждого нового Pod
- Автоматический rollback при проблемах


**Преимущества:**
- Нулевой downtime для stateless сервисов
- Автоматический rollback
- Простой в настройке

**Ограничения:**
- Не работает для stateful сервисов
- Требует совместимости версий API

===== Canary Deployments

**Принцип работы:**
- Развертывание новой версии на небольшом подмножестве пользователей
- Мониторинг метрик и ошибок
- Постепенное увеличение трафика на новую версию

**Инструменты:**
- Argo Rollouts
- Flagger
- Istio


**Метрики для анализа:**
- Error rate
- Latency (p95, p99)
- CPU/Memory utilization
- User engagement

**Преимущества:**
- Минимизация риска для пользователей
- Быстрое обнаружение проблем
- Постепенный rollback

===== Blue-Green Deployments

**Принцип работы:**
- Две идентичные среды: Blue (текущая) и Green (новая)
- Полное развертывание новой версии в Green
- Переключение трафика с Blue на Green
- Blue остается для быстрого rollback

**Инструменты:**
- Kubernetes Services
- Istio VirtualService
- Nginx Ingress


**Преимущества:**
- Мгновенный переключение трафика
- Полная изоляция версий
- Быстрый rollback

**Ограничения:**
- Требует в 2 раза больше ресурсов
- Сложнее в управлении

===== Database Migrations

**Проблемы:**
- Изменения схемы БД могут вызвать downtime
- Несовместимость между версиями приложения и БД

**Стратегии:**

**1. Expand-Contract Pattern:**
- **Expand:** Добавить новую колонку/таблицу (без удаления старой)
- **Deploy:** Развернуть приложение, работающее с обеими версиями
- **Migrate:** Перенести данные (в фоне)
- **Contract:** Удалить старую колонку/таблицу



**2. Versioned Migrations:**
- Каждая миграция имеет версию
- Миграции выполняются в порядке версий
- Возможность rollback

**Инструменты:**
- Flyway
- Liquibase


**Процесс:**
1. Создать миграцию в feature branch
2. Code Review миграции
3. Тестирование миграции на dev/staging
4. Выполнить миграцию перед деплоем приложения
5. Разворачивание приложения
6. Мониторинг

**Когда downtime неизбежен:**
- Крупные изменения схемы БД
- Миграция больших объемов данных
- Изменения, требующие exclusive lock
- Обновление версии PostgreSQL

**Минимизация downtime:**
- Планирование в периоды низкой активности
- Использование read replicas для чтения
- Подготовка rollback плана
- Коммуникация с пользователями

==== 5. Rollback стратегии

===== Автоматический Rollback

**Kubernetes:**
- Deployment автоматически откатывается при неудачном обновлении
- Liveness и readiness probes определяют проблемы
- Настройка `revisionHistoryLimit` для хранения истории

**ArgoCD:**
- Автоматический sync при отклонении от Git
- Auto-sync можно отключить для production
- Manual approval для критических изменений

===== Ручной Rollback

**Через ArgoCD:**
1. Выбрать приложение в ArgoCD UI
2. Перейти в History
3. Выбрать предыдущую ревизию
4. Нажать Rollback



**Database Rollback:**
- Подготовка downgrade миграций
- Тестирование rollback на staging
- Выполнение rollback миграций
- Развертывание предыдущей версии приложения

==== 6. Мониторинг деплоев

===== Метрики деплоя

**Deployment Frequency:**
- Количество деплоев в день/неделю
- Цель: 1-2 деплоя в день

**Lead Time:**
- Время от commit до production
- Цель: < 1 часа для hotfix, < 1 дня для feature

**Change Failure Rate:**
- Процент деплоев, требующих rollback
- Цель: < 5%

**Mean Time to Recovery (MTTR):**
- Время восстановления после сбоя
- Цель: < 1 часа

===== Алертинг

**Алерты при деплое:**
- Error rate > 1% после деплоя
- Latency > 2x от baseline
- Pod restart rate > 5%
- Database connection errors

**Действия при алертах:**
- Автоматический rollback при критических ошибках
- Уведомление команды в мессенджере
- Анализ логов и метрик
- Принятие решения о rollback или продолжении

==== 7. Безопасность CI/CD

===== Secrets Management

**Хранение секретов:**
- GitHub Secrets / GitLab Variables
- HashiCorp Vault для production
- Отдельные секреты для каждого окружения

**Доступ к секретам:**
- Минимальные привилегии
- Ротация секретов
- Аудит доступа
