=== Форматы взаимодействия

В этом разделе описаны форматы данных и протоколы, используемые для взаимодействия между компонентами системы.

==== REST API (Swagger/OpenAPI)

*Применение:* Клиент <> API Gateway, внешние интеграции
*Протокол:* HTTPS (HTTP/1.1, HTTP/2)
*Формат:* JSON
*Документация:* OpenAPI 3.0, доступна по /swagger
*Пример запроса:*

[source,json]
----
POST /api/auth/login
Content-Type: application/json

{
  "login": "user@example.com",
  "otp": "123456"
}
----

*Пример ответа:*

[source,json]
----
{
  "jwt_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_at": 1774853600,
  "user_id": "uuid-1234-5678-90ab-cdef12345678"
}
----

==== gRPC (Protobuf)

*Применение:* API Gateway <> Сервисы (внутренняя коммуникация)
*Протокол:* gRPC over HTTP/2
*Формат:* Protobuf (бинарный формат)
*Преимущества:* Высокая производительность, строгая типизация, автоматическая генерация кода
*Пример .proto файла:*

[source,protobuf]
----
syntax = "proto3";

package chat;

message SendMessageRequest {
  string chat_id = 1;
  string text = 2;
  string media_id = 3;
}

message SendMessageResponse {
  string message_id = 1;
  int64 sent_at = 2;
}
----

==== WebSocket

*Применение:* Клиент <> API Gateway (реальное время)
*Протокол:* WebSocket (WS/WSS)
*Формат:* JSON
*Использование:* Подписка на новые сообщения, статусы набора текста, онлайн-статусы
*Пример сообщения:*

[source,json]
----
{
  "type": "message",
  "chat_id": "uuid-1234",
  "message": {
    "message_id": "uuid-5678",
    "sender_id": "uuid-90ab",
    "text": "Привет!",
    "sent_at": 1774853600
  }
}
----

==== Kafka (Брокер сообщений)

*Применение:* Асинхронная коммуникация между сервисами
*Протокол:* TCP (PLAINTEXT/TLS)
*Формат:* Protobuf/Avro/JSON
*Schema Registry:* Confluent Schema Registry для управления схемами
*Пример события:*

[source,protobuf]
----
syntax = "proto3";

package events;

message MessageCreatedEvent {
  string message_id = 1;
  string chat_id = 2;
  string sender_id = 3;
  string text = 4;
  string media_id = 5;
  int64 sent_at = 6;
}
----

*Топики Kafka:*
* `chat.message.created` - новые сообщения
* `chat.message.deleted` - удаленные сообщения
* `chat.group.created` - создание групповых чатов
* `media.uploaded` - загрузка медиа
* `user.blocked` - блокировка пользователя
* `pg.changes` - CDC события от Debezium

==== Outbox Pattern - Формат событий

*Применение:* Обеспечение exactly-once семантики при публикации в Kafka
*Протокол:* Kafka (TCP)
*Формат:* JSON/Protobuf

*Таблица outbox в PostgreSQL:*

[source,sql]
----
CREATE TABLE outbox (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(50) NOT NULL,  -- 'message', 'user', 'chat'
    aggregate_id UUID NOT NULL,           -- message_id, user_id, chat_id
    event_type VARCHAR(100) NOT NULL,     -- 'MessageCreated', 'UserUpdated'
    payload JSONB NOT NULL,               -- данные события
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    processed BOOLEAN DEFAULT FALSE,
    processed_at TIMESTAMP
);

CREATE INDEX idx_outbox_processed ON outbox(processed, created_at);
----

*Пример события в outbox:*

[source,json]
----
{
  "id": "uuid-outbox-1234",
  "aggregate_type": "message",
  "aggregate_id": "uuid-message-5678",
  "event_type": "MessageCreated",
  "payload": {
    "message_id": "uuid-message-5678",
    "chat_id": "uuid-chat-9012",
    "sender_id": "uuid-user-3456",
    "text": "Привет, как дела?",
    "sent_at": 1774853600
  },
  "created_at": "2024-01-01T12:00:00Z",
  "processed": false,
  "processed_at": null
}
----

*Процесс обработки outbox:*
1. Транзакционно записываем бизнес-данные и событие в outbox
2. Debezium читает из outbox и публикует в Kafka
3. Consumer обрабатывает событие с идемпотентной логикой



*Формат подтверждения от Consumer:*

[source,json]
----
{
  "outbox_id": "uuid-outbox-1234",
  "aggregate_type": "message",
  "aggregate_id": "uuid-message-5678",
  "event_type": "MessageCreated",
  "consumer_id": "search-service",
  "processed_at": "2024-01-01T12:05:00Z",
  "status": "PROCESSED"
}
----

*Описание полей подтверждения:*
- `outbox_id` - идентификатор записи в outbox таблице
- `aggregate_type` - тип агрегата (message, user, chat)
- `aggregate_id` - идентификатор агрегата
- `event_type` - тип события
- `consumer_id` - идентификатор сервиса-потребителя
- `processed_at` - время успешной обработки
- `status` - статус обработки (PROCESSED, FAILED)


==== Семантика доставки сообщений

*At-least-once (минимум один раз):*

*Характеристики:*
- Kafka гарантирует доставку сообщений минимум один раз
- Возможны дубликаты сообщений
- Используется для некритичных операций (уведомления, индексация)

*Обработка дубликатов:*
- Дедупликация по message_id или event_id
- Идемпотентные операции (upsert вместо insert)
- Логирование дубликатов для мониторинга

*Пример реализации:*

[source,protobuf]
----
syntax = "proto3";

package events;

message MessageCreatedEvent {
  string message_id = 1;  // используется для дедупликации
  string chat_id = 2;
  string sender_id = 3;
  string text = 4;
  string media_id = 5;
  int64 sent_at = 6;
  int64 event_timestamp = 7;  // для отслеживания порядка
}
----

*Exactly-once (ровно один раз):*

*Характеристики:*
- Гарантирует, что сообщение будет обработано ровно один раз
- Используется для критичных операций (создание сообщений, пользователей)
- Требует дополнительных механизмов для обеспечения

*Механизмы обеспечения exactly-once:*

1. **Outbox Pattern:**
   - Транзакционная запись в PostgreSQL + публикация в Kafka
   - Debezium читает из outbox и публикует в Kafka
   - Consumer обрабатывает событие с идемпотентной логикой

2. **Идемпотентные операции:**
   - Использование уникальных ключей (message_id, event_id)
   - Upsert операции вместо insert
   - Проверка существования перед обработкой

3. **Дедупликация:**
   - Хранение обработанных event_id в Redis или PostgreSQL
   - TTL для очистки старых записей
   - Проверка перед обработкой

*Пример реализации exactly-once:*

[source,protobuf]
----
syntax = "proto3";

package events;

message MessageCreatedEvent {
  string message_id = 1;  // уникальный идентификатор
  string chat_id = 2;
  string sender_id = 3;
  string text = 4;
  string media_id = 5;
  int64 sent_at = 6;
  string event_id = 7;  // уникальный идентификатор события для дедупликации
  int64 event_timestamp = 8;
}




==== Базы данных

*PostgreSQL:*
*Протокол:* TCP
*Формат:* SQL + бинарные данные
*Использование:* Хранение структурированных данных (пользователи, чаты, сообщения)

*Redis:*
*Протокол:* TCP
*Формат:* Key-Value (строки, хэши, списки)
*Использование:* Кэширование, сессии, очереди

*Elasticsearch:*
*Протокол:* HTTP (REST)
*Формат:* JSON
*Использование:* Полнотекстовый поиск, индексация сообщений

==== Внешние интеграции

*LDAP (Active Directory):*
*Протокол:* LDAP + TLS
*Формат:* LDIF
*Использование:* Синхронизация пользователей, проверка учетных данных


*SMTP (Email):*
*Протокол:* SMTP + TLS
*Формат:* MIME
*Использование:* Отправка OTP, уведомлений по почте

Swagger/OpenAPI документация

Документация REST API доступна по адресу `/swagger` и включает:
* Все публичные эндпоинты
* Схемы запросов и ответов
* Примеры использования
* Информацию об аутентификации (JWT)

Пример OpenAPI спецификации:

[source,yaml]
----
openapi: 3.0.0
info:
  title: F.U.C.K. Messenger API
  version: 1.0.0
  description: API для корпоративного мессенджера

servers:
  - url: https://api.messenger.company.com/v1
    description: Production server

paths:
  /auth/login:
    post:
      summary: Аутентификация пользователя
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              properties:
                login:
                  type: string
                otp:
                  type: string
      responses:
        '200':
          description: Успешная аутентификация
          content:
            application/json:
              schema:
                type: object
                properties:
                  jwt_token:
                    type: string
                  expires_at:
                    type: integer
                  user_id:
                    type: string
        '401':
          description: Неверные учетные данные
----
