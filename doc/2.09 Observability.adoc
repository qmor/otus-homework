=== Observability
схема мониторинга и логирования

. Необходимо собирать логи со всех сервисов в ELK стек (Elasticsearch, Logstash, Kibana)
. Все сервисы должны писать метрики в Prometheus через exporters
. Используем Grafana для отображения метрик и создания дашбордов
. Используем OpenSearch для отображения логов и их анализа
. Используем Jaeger/Zipkin для распределенной трассировки (Distributed Tracing)
. Alertmanager для отправки оповещений (Slack, Email, SMS)



какие метрики планируется собирать

*Производительность:*
. Время отклика API (p50, p95, p99 перцентили)
. Время выполнения запроса в БД
. Задержка передачи сообщения
. Количество запросов в секунду (RPS)
. Количество сообщений в секунду

*Нагрузка:*
. Количество активных пользователей (DAU, MAU)
. Использование CPU, RAM, диска
. Количество открытых соединений
. Размер топиков в Kafka

*Бизнес-метрики:*
. Количество созданных чатов
. Количество отправленных сообщений
. Количество загруженных медиа-файлов
. Скорость увеличения объема медиа контента

*Надежность:*
. Количество ошибок (5xx/4xx)
. Uptime сервисов
. Время и статус последнего бэкапа
. Количество сообщений в DLQ (Dead Letter Queue)
. Количество роликов в очереди на конвертацию
. Количество изображений в очереди на сохранение

*Безопасность:*
. Количество неудачных попыток авторизации
. Количество заблокированных пользователей
. Количество подозрительных запросов (аномальная активность)

SLI/SLO

==== Service Level Objectives (SLO)

*Окно измерения:* 30 дней rolling window

*Availability:*
- SLI: % успешных 2xx ответов
- SLO: 99.9%
- Error Budget: 0.1% = ~43.2 минуты простоя в месяц
- Измерение: rolling window за 30 дней

*Latency:*
- SLI: 95-й перцентиль времени отклика API
- SLO: < 500 мс для текстовых сообщений
- Error Budget: превышение 500 мс считается ошибкой
- Измерение: rolling window за 30 дней

*Error Rate:*
- SLI: % запросов с ошибками (4xx + 5xx)
- SLO: < 0.1%
- Error Budget: 0.1% = ~43.2 минуты простоя в месяц
- Измерение: rolling window за 30 дней

==== Error Budget Policy

*Расчет Error Budget:*
- 30 дней = 43,200 минут
- Error Budget 0.1% = 43.2 минуты простоя в месяц
- При достижении 50% error budget: P2 алерты (предупреждающие)
- При достижении 90% error budget: P1 алерты (критичные)
- При исчерпании error budget: заморозка релизов, приоритет исправлений

==== Связь SLO с алертингом

*P1 алерты (критичные - немедленное реагирование):*
- API Gateway: 5xx ошибок > 5% (угроза SLO Availability)
- Сервис недоступен (Health Check failed) (угроза SLO Availability)
- Kafka: лаги в топиках > 30 секунд (угроза SLO Latency)
- PostgreSQL: CPU > 90% или диск заполнен > 85% (угроза SLO Availability)
- Redis: использование памяти > 90% (угроза SLO Availability)
- Количество сообщений в DLQ > 100 (угроза SLO Error Rate)
- Error Budget израсходован на 90% (критическое состояние)

*P2 алерты (предупреждающие - реагирование в течение часа):*
- Задержка API > 1 секунды (p95) (угроза SLO Latency)
- Увеличение количества ошибок в логах > 1% (угроза SLO Error Rate)
- Несостоявшийся вовремя бэкап (угроза SLO Availability)
- Превышение скорости увеличения объема медиа контента (угроза SLO Latency)
- Error Budget израсходован на 50% (предупреждающее состояние)

*P3 алерты (информационные - мониторинг):*
- Высокая нагрузка на Kafka
- Увеличение количества активных пользователей
- Близкое к заполнению хранилище S3
- Error Budget израсходован на 25% (информационное состояние)

==== Burn Rate Alerting

*Burn Rate - скорость расходования error budget:*
- Burn Rate 1x: нормальное состояние, error budget расходуется равномерно
- Burn Rate 2x: error budget расходуется в 2 раза быстрее нормы (P2 алерт)
- Burn Rate 5x: error budget расходуется в 5 раз быстрее нормы (P1 алерт)
- Burn Rate 10x: критическая ситуация, немедленное реагирование (P1 алерт)

*Пример расчета:*
- При SLO 99.9% и burn rate 5x, error budget будет израсходован за 6 дней вместо 30
- Это требует немедленного вмешательства для предотвращения нарушения SLO
