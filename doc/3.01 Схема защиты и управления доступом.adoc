=== Схема защиты и управления доступом

==== Общий подход

Система использует многоуровневую защиту:

*Транспортный уровень:*
- Все соединения между клиентами и API Gateway защищены TLS 1.3
- Внутренняя коммуникация между сервисами защищена mTLS (mutual TLS)
- Шифрование данных при передаче через все протоколы (HTTPS, gRPC, WebSocket)

*Уровень хранения:*
- Шифрование данных в покое (at-rest encryption) для всех хранилищ
- PostgreSQL: шифрование на уровне файловой системы или базы данных
- S3/MinIO: шифрование на стороне сервера (SSE-S3 или SSE-KMS)
- Redis: шифрование данных в памяти (опционально, для чувствительных данных)

*Важно:* Система НЕ реализует end-to-end шифрование (E2EE) сообщений. Мессенджер корпоративный такое шифрование не требуется. Это позволяет:
- Осуществлять полнотекстовый поиск по содержимому сообщений на сервере
- Реализовать модерацию и фильтрацию контента
- Обеспечивать соответствие корпоративным требованиям к аудиту и хранению данных


==== 1. Аутентификация и авторизация
*Аутентификация:*
- Интеграция с корпоративным LDAP (Active Directory)
- Двухфакторная аутентификация через OTP (One-Time Password)
- JWT токены для сессий пользователей
- Время жизни токена: 1 час (refresh token: 30 дней)

===== JWT токены

Используем JWT для аутентификации.

- Хорошо подходит для микросервисов

**Процесс входа:**
1. Пользователь вводит email
2. Система отправляет OTP на email
3. Пользователь вводит OTP
4. Auth Service проверяет через LDAP
5. Генерируется JWT токен
6. Токен возвращается пользователю

*Авторизация:*
- RBAC (Role-Based Access Control)
- Роли: Admin, Moderator, User
- Проверка прав доступа на уровне API Gateway и сервисов
- Контроль доступа к чатам и каналам

**Реализация:**
- Таблицы roles, permissions, role_permissions, user_roles в PostgreSQL
- Middleware для проверки прав в каждом сервисе

===== mTLS для сервисов

Каждый сервис имеет свой сертификат и проверяет сертификаты других сервисов.

==== 2. Безопасность передачи данных

===== TLS 1.3

Все данные передаются по защищённому каналу.

===== HSTS

Заголовок, который заставляет браузер всегда использовать HTTPS.

===== Шифрование данных

Даже при TLS шифруем чувствительные данные:
- Персональные данные
- Содержимое сообщений
- Медиа-файлы

**Алгоритм:** AES-256-GCM

==== 3. Хранение секретов


Инструмент для безопасного хранения секретов  HashiCorp Vault.

- Пароли баз данных
- JWT ключи
- Ключи шифрования
- API ключи сервисов


**Интеграция с Kubernetes:**
- Vault Agent автоматически внедряет секреты в Pods
- Секреты доступны как переменные окружения

==== 4. Защита от атак

===== Rate Limiting

Ограничение количества запросов:
- 100 req/sec на IP
- 1000 req/sec на пользователя
- Блокировка при превышении

===== Защита от инъекций

- Параметризованные запросы к БД
- Валидация входных данных
- Защита от XSS и CSRF

===== Мониторинг безопасности

- Логирование всех событий безопасности
- Детекция аномальной активности
- Автоматические оповещения при подозрительных действиях

==== 5. Соответствие требованиям

===== Корпоративные требования
- Хранение логов всех действий пользователей
- Возможность экспорта данных для аудита
- Контроль доступа к чувствительным данным
- Соответствие политикам информационной безопасности предприятия

===== GDPR
- Право на удаление данных
- Минимизация собираемых данных
- Защита персональных данных

==== 6. Резервное копирование

===== Стратегия бэкапов
- Ежедневные инкрементальные бэкапы
- Еженедельные полные бэкапы
- Хранение бэкапов в зашифрованном виде

===== Восстановление
- Время восстановления (RTO): 4 часа
- Точка восстановления (RPO): 1 час
- Документированные playbooks для сценариев инцидентов
