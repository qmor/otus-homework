=== Схема защиты и управления доступом

==== Общий подход

Система использует многоуровневую защиту:

*Транспортный уровень:*
- Все соединения между клиентами и API Gateway защищены TLS 1.3
- Внутренняя коммуникация между сервисами защищена mTLS (mutual TLS)
- Шифрование данных при передаче через все протоколы (HTTPS, gRPC, WebSocket)

*Уровень хранения:*
- Шифрование данных в покое (at-rest encryption) для всех хранилищ
- PostgreSQL: шифрование на уровне файловой системы или базы данных
- S3/MinIO: шифрование на стороне сервера (SSE-S3 или SSE-KMS)
- Redis: шифрование данных в памяти (опционально, для чувствительных данных)

*Важно:* Система НЕ реализует end-to-end шифрование (E2EE) сообщений. Мессенджер корпоративный такое шифрование не требуется. Это позволяет:
- Осуществлять полнотекстовый поиск по содержимому сообщений на сервере
- Реализовать модерацию и фильтрацию контента
- Обеспечивать соответствие корпоративным требованиям к аудиту и хранению данных


==== 1. Аутентификация и авторизация
*Аутентификация:*
- Интеграция с корпоративным LDAP (Active Directory)
- Двухфакторная аутентификация через OTP (One-Time Password)
- JWT токены для сессий пользователей
- Время жизни токена: 1 час (refresh token: 30 дней)

===== JWT токены

Используем JWT для аутентификации.

- Хорошо подходит для микросервисов

**Процесс входа:**
1. Пользователь вводит email
2. Система отправляет OTP на email
3. Пользователь вводит OTP
4. Auth Service проверяет через LDAP
5. Генерируется JWT токен
6. Токен возвращается пользователю

===== Выбор механизмов аутентификации и авторизации

Для разных компонентов системы используются разные механизмы:

**Для пользователей (веб и мобильные приложения):**
- **JWT токены** - основной механизм для аутентификации пользователей
- **LDAP** - для проверки учетных данных в корпоративном каталоге

**Для API между сервисами:**
- **mTLS** - для безопасной коммуникации между микросервисами
- Каждый сервис имеет свой сертификат и проверяет сертификаты других сервисов
- Это гарантирует, что только авторизованные сервисы могут общаться друг с другом



- JWT - легковесный, не требует хранения состояния на сервере, идеально для микросервисов
- mTLS - самый надежный способ защиты сервис-сервис коммуникации
- LDAP - стандарт для корпоративной аутентификации

===== Выдача и проверка токенов доступа

**Как выдаются токены:**

1. **Вход пользователя:**
   - Пользователь вводит email и получает OTP
   - Auth Service проверяет OTP и учетные данные в LDAP
   - Если всё верно, генерируется пара токенов:
     - *Access Token*  - короткоживущий токен для доступа к API
     - *Refresh Token* - долгоживущий токен для получения новых access токенов


2. **Хранение токенов:**
   - Access Token хранится в памяти браузера/приложения
   - Refresh Token хранится в защищенном хранилище (HttpOnly cookie или secure storage)

**Как проверяются токены:**

1. **На API Gateway:**
   - Проверяется наличие токена в заголовке Authorization
   - Проверяется цифровая подпись токена
   - Проверяется срок действия токена
   - Извлекается информация о пользователе и ролях

2. **В микросервисах:**
   - Токен передается в заголовке каждого запроса
   - Каждый сервис может проверить токен самостоятельно
   - Проверяются права доступа на основе ролей из токена

3. **Проверка Refresh Token:**
   - Refresh Token проверяется в Auth Service
   - Проверяется, что токен не отозван и не истек
   - При успешной проверке выдается новая пара токенов

===== Архитектура управления токенами

**Сроки действия токенов:**
- Access Token: 1 час
- Refresh Token: 30 дней
- Refresh Token можно использовать только один раз

**Процесс обновления токенов:**

1. **Автоматическое обновление:**
   - Клиентское приложение отслеживает срок действия access токена
   - За 5 минут до истечения токена отправляется запрос на обновление
   - Auth Service проверяет refresh токен и выдает новую пару токенов
   - Старый refresh токен аннулируется

2. **Сценарий истечения access токена:**
   - Если access токен истек, клиент получает ошибку 401
   - Клиент использует refresh токен для получения новой пары
   - Если refresh токен тоже истек, пользователь должен войти заново

3. **Отзыв токенов:**
   - При выходе пользователя refresh токен добавляется в черный список
   - При смене пароля все токены пользователя аннулируются
   - При подозрительной активности все токены могут быть отозваны

**Хранение отозванных токенов:**
- Используется Redis для хранения черного списка refresh токенов
- Токены хранятся в списке до истечения их срока действия
- Это позволяет быстро проверить, не был ли токен отозван

**Безопасность токенов:**
- Все токены подписываются секретным ключом из HashiCorp Vault
- Ключи регулярно ротируются (например, раз в 90 дней)
- При ротации ключей старые токены остаются валидными до истечения срока
- Новые токены подписываются новым ключом

**Обработка ошибок:**
- Неверный токен → ошибка 401 Unauthorized
- Истекший токен → ошибка 401 с указанием, что нужно обновить
- Отозванный токен → ошибка 401
- Отсутствие токена → ошибка 401 для защищенных ресурсов

**Мониторинг токенов:**
- Логирование всех операций с токенами (выдача, обновление, отзыв)
- Мониторинг количества неудачных попыток использования токенов
- Оповещения при подозрительной активности с токенами

*Авторизация:*
- RBAC (Role-Based Access Control)
- Роли: Admin, Moderator, User
- Проверка прав доступа на уровне API Gateway и сервисов
- Контроль доступа к чатам и каналам

**Реализация:**
- Таблицы roles, permissions, role_permissions, user_roles в PostgreSQL
- Middleware для проверки прав в каждом сервисе

===== mTLS для сервисов

Каждый сервис имеет свой сертификат и проверяет сертификаты других сервисов.

==== 2. Безопасность передачи данных

===== TLS 1.3

Все данные передаются по защищённому каналу.

===== HSTS

Заголовок, который заставляет браузер всегда использовать HTTPS.

===== Шифрование данных

Даже при TLS шифруем чувствительные данные:
- Персональные данные
- Содержимое сообщений
- Медиа-файлы

**Алгоритм:** AES-256-GCM

==== 3. Хранение секретов


Инструмент для безопасного хранения секретов  HashiCorp Vault.

- Пароли баз данных
- JWT ключи
- Ключи шифрования
- API ключи сервисов


**Интеграция с Kubernetes:**
- Vault Agent автоматически внедряет секреты в Pods
- Секреты доступны как переменные окружения

==== 4. Защита от атак

===== Rate Limiting

Ограничение количества запросов:
- 100 req/sec на IP
- 1000 req/sec на пользователя
- Блокировка при превышении

===== Защита от инъекций

- Параметризованные запросы к БД
- Валидация входных данных
- Защита от XSS и CSRF

===== Мониторинг безопасности

- Логирование всех событий безопасности
- Детекция аномальной активности
- Автоматические оповещения при подозрительных действиях

==== 5. Соответствие требованиям

===== Корпоративные требования
- Хранение логов всех действий пользователей
- Возможность экспорта данных для аудита
- Контроль доступа к чувствительным данным
- Соответствие политикам информационной безопасности предприятия

===== GDPR
- Право на удаление данных
- Минимизация собираемых данных
- Защита персональных данных

==== 6. Резервное копирование

===== Стратегия бэкапов
- Ежедневные инкрементальные бэкапы
- Еженедельные полные бэкапы
- Хранение бэкапов в зашифрованном виде

===== Восстановление
- Время восстановления (RTO): 4 часа
- Точка восстановления (RPO): 1 час
- Документированные playbooks для сценариев инцидентов
